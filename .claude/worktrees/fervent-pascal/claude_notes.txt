# Multi-Issue Photo Assignment - Session Summary (Apr 3, 2026)

## Task
Implement intelligent photo assignment for split multi-issue flow. Previously, when multiple issues were detected and split, ALL photos were uploaded only to the first issue. Now users can assign individual photos to each split issue during confirmation.

## Changes Made

### 1. Updated `lib/ai/detect-multi-issue.ts`
- **Added Field**: `photoHint?: string` to `singleIssueSchema`
- **Purpose**: AI generates a brief, user-friendly hint about what type of photo is relevant to each issue
- **Example Values**:
  - "photo of leaking pipe"
  - "photo of broken AC unit"
  - "photo of window damage"
- **Location in Schema**: Zod schema updated to include optional photoHint field
- **Updated Type Exports**: `DetectedIssue` type now includes optional `photoHint`
- **AI Prompt Update**: Added instruction to AI:
  ```
  photoHint: a brief description of what kind of photo would be most relevant
  to this issue (e.g., "photo of leaking pipe" or "photo of broken AC unit").
  This helps the user assign their photos correctly. Keep it short (under 20 words)
  and specific.
  ```

### 2. Updated `app/(app)/issues/new/page.tsx` - Main Functionality

#### 2a. Added Photo Assignment State
- **New State**: `photoAssignments: Record<number, Set<string>>({})`
- **Purpose**: Maps issue index to Set of assigned photo IDs
- **Initialization**: Happens when multi-issue confirmation screen first renders

#### 2b. Added Helper Functions
- **togglePhotoAssignment(issueIndex, photoId)**:
  - Toggles a photo's assignment to an issue
  - Updates state using Set operations for O(1) lookup
  - Handles creating new Sets for issues if they don't exist

- **initializePhotoAssignments()**:
  - Called when split confirmation screen renders
  - Only initializes if `photoAssignments` is empty (prevents reinitializing on every render)
  - Pre-checks all photos for all issues by default (good UX - user sees everything assigned)
  - Creates Set for each issue with all photo IDs

#### 2c. Updated `uploadPhotos()` Function
- **New Parameter**: `photoIds?: Set<string>`
- **Logic**:
  - If `photoIds` provided, filters photos to only those with IDs in the Set
  - If no `photoIds` (single-issue flow), uploads all photos as before
  - Maintains backward compatibility

#### 2d. Updated `handleConfirmSplit()` Function
- **Key Change**: Now uses indexed loop `for (let i = 0; i < detectedIssues.length; i++)`
- **Photo Upload Logic**:
  ```
  if (photos.length > 0) {
    const assignedPhotos = photoAssignments[i];
    if (assignedPhotos && assignedPhotos.size > 0) {
      await uploadPhotos(created.id, assignedPhotos);
    }
  }
  ```
- **Behavior**: Only uploads photos that were explicitly checked for each issue
- **Previous Behavior**: Uploaded all photos only to the first issue

#### 2e. Updated Split Confirmation UI
- **Location**: Multi-issue split confirmation screen (renders when `detectedIssues.length > 1`)
- **Call**: `initializePhotoAssignments()` at start of render to ensure state is initialized
- **For Each Issue Card**:
  - Shows `issue.photoHint` if available (renders as italic gray text below description)
  - Format: "Suggested photo: {photoHint}"
  - Added photo assignment section below issue details (only shown if photos exist)

- **Photo Assignment UI**:
  ```
  {photos.length > 0 && (
    <div className="pt-2 border-t">
      <label>Assign photos to this issue</label>
      <div className="grid grid-cols-3 gap-2">
        {photos.map((photo) => (
          <button>
            <Image src={photo.preview} />
            {isAssigned && (
              <CheckmarkIcon /> // Shows checkmark overlay
            )}
          </button>
        ))}
      </div>
    </div>
  )}
  ```

- **Visual Design**:
  - Grid layout with 3 columns
  - Clickable photo thumbnails (h-20 height)
  - Border color indicates assignment state:
    - `border-primary bg-primary/5` when assigned
    - `border-border bg-muted/50` when not assigned
  - Checkmark icon (SVG path) overlays assigned photos with semi-transparent black background
  - Smooth transitions on hover/click

### 3. Updated Interface Definition
- **DetectedIssue Interface** in page.tsx now includes `photoHint?: string`

## User Flow

1. User describes multiple issues (e.g., "toilet leak and broken AC")
2. AI detects 2 issues and returns:
   - Issue 1: "toilet leak" with photoHint: "photo of leaking toilet"
   - Issue 2: "broken AC" with photoHint: "photo of AC unit"
3. Confirmation screen shows:
   - Both issues with their descriptions and hints
   - All uploaded photos in a 3-column grid under each issue
   - All photos pre-checked for each issue (default: share context)
4. User can uncheck photos that don't apply to specific issues
5. When creating issues:
   - Issue 1 gets photos user kept checked
   - Issue 2 gets photos user kept checked
   - Photos not assigned to any issue won't be uploaded

## Design Decisions

1. **AI Can't See Photos**: Since AI only has text description at detection time, we can't do vision-based assignment. Instead, AI provides helpful hints to guide user selection.

2. **All Assigned By Default**: Pre-checks all photos for each issue because split issues often represent different aspects of the same problem that all photos help document.

3. **Set Data Structure**: Uses `Set<string>` for O(1) lookup performance when checking if photo is assigned.

4. **Default Behavior Maintained**: Single-issue flow (non-split) unchanged - all photos still uploaded to the single issue.

5. **Photo Hints as Guidance**: Not mandatory assignment rules - user can override hints if they prefer. Hints are just suggestions to improve UX.

6. **No Drag-and-Drop**: Simpler checkbox-based UI instead of complex drag-and-drop that could confuse users and add complexity.

# Cost Intelligence & Recurring Issue Detection - Session Summary (Apr 3, 2026)

Build two intelligent features for Maintenance OS that provide cost context and detect recurring problems at properties.

## Feature 1: Cost Intelligence

### API Endpoint: `app/api/issues/[id]/cost-context/route.ts`
**Purpose**: Return historical cost data for issues with the same category at the same property

**Implementation**:
- Uses `requireDbUser()` for authentication
- Finds current issue to verify ownership and extract category/propertyId
- Queries completed jobs where:
  - Issue has same category
  - Issue is on same property
  - Job status is completed and actualCost is not null
- Calculates: min cost, max cost, average cost, count of historical repairs
- Returns null if no history exists (graceful no-render pattern)

**Query Pattern**:
```typescript
const historicalJobs = await prisma.job.findMany({
  where: {
    issue: {
      propertyId: issue.propertyId,
      category: issue.category,
      status: 'completed',
    },
    status: 'completed',
    actualCost: { not: null },
  },
  select: { actualCost: true },
});
```

## Feature 2: Recurring Issue Detection

### API Endpoint: `app/api/issues/[id]/patterns/route.ts`
**Purpose**: Detect if property has repeated issues of same category

**Implementation**:
- Uses `requireDbUser()` for authentication
- Finds current issue to verify ownership and extract propertyId/category
- Counts similar issues in past 12 months:
  - Same propertyId
  - Same category
  - Excluding current issue
  - Created >= 1 year ago
- Returns pattern data only if count >= 2 (making current issue the 3rd+)
- Includes `previousIssueDates` for context

**Query Pattern**:
```typescript
const oneYearAgo = new Date(issue.createdAt.getTime() - 365 * 24 * 60 * 60 * 1000);
const similarIssuesCount = await prisma.issue.count({
  where: {
    propertyId: issue.propertyId,
    category: issue.category,
    id: { not: id },
    createdAt: { gte: oneYearAgo },
  },
});
```

**Features**:
- Accepts props: issueId, category, propertyName
- Fetches pattern data from API on mount
- Shows amber alert banner only if pattern detected (3+ issues):
  - Format: "This is the 3rd [category] issue at [property] this year."
  - Subtitle: "Consider scheduling a professional inspection to address the root cause."
- Gracefully renders nothing if no pattern exists
- Handles ordinal numbers: "3rd", "4th", "5th", etc.

**Design**:
- Amber background banner with ! icon
- Suggests proactive inspection
- Matches recurring problem warning UI pattern

### Integration

- Early in the issue detail page (before other content)
- Only renders if recurring pattern is detected (no noise)

### 1. Manual Quote Dedupe (app/api/issues/[id]/manual-quote/route.ts)
**Problem**: No check for existing manual quote on same dispatch. Double-click creates duplicate contractorResponse rows.
**Fix**: Inside the transaction, before creating a new response, check for an existing `[Manual entry]` response on the same dispatch. If found, return it instead of creating a duplicate. This is idempotent — the second request gets the same result as the first.

### 2. Reply-to-Contractor Dedupe (app/api/issues/[id]/reply-to-contractor/route.ts)
**Problem**: No duplicate send protection. Double-click sends the same message twice to the contractor.
**Fix**: Before creating the message record, check for a recent (within 60 seconds) outbound message with the same body to the same contractor on the same issue, with status 'queued' or 'sent'. If found, return success without sending again. Window is short enough to not block intentional re-sends.
