Here's my take on each:
Should dispatching after quotes_received be allowed? Yes, keep it. In real life, a property manager gets two quotes back, neither is great, and they want to send it to one more contractor. Blocking that would be frustrating. The code already handles it cleanly — the race-safe updateMany won't regress the status backwards. Just make sure the UI makes it obvious they're adding more contractors to an issue that already has quotes.
Should classified and awaiting_dispatch be merged? Yes, merge them. Right now they're visually identical (same label, same color) and there's no user action that moves an issue from classified to awaiting_dispatch — it's a phantom state. Every route that checks one also checks the other. Kill awaiting_dispatch, just use classified to mean "AI looked at it, ready to send to contractors." One less status to maintain, one less thing to confuse users. This is a schema migration though, so it's not a 5-minute fix — you'd need to update the Prisma enum, migrate existing rows, and update every reference. I'd do it, but not today.
Should manual-quote work from new? Yes, but with a caveat. Sometimes a property manager already knows who they want and what the price is — they talked to the contractor on the phone and just want to log it. Forcing them to classify first adds friction for no reason. The issue will get a title and category from the manual entry context anyway. Keep it allowed.
So the short version: keep dispatching after quotes, keep manual-quote from new, and merge classified/awaiting_dispatch when you're ready for a migration. None of these are urgent — the code handles all three scenarios correctly right now. The merge is the only one worth scheduling, and it's a cleanup, not a fix.
