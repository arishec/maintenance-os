MAINTENANCE OS
HIRING + JOB LIFECYCLE SPEC
(CLEANLY PLUGGED INTO THE SYSTEM YOU ALREADY BUILT)

==================================================
PURPOSE
==================================================

This document defines the missing core workflow after quote collection:

QUOTE RECEIVED → SELECT CONTRACTOR → CREATE JOB → TRACK JOB → COMPLETE JOB

It is designed to plug into what you already have:

Already built:
- issues
- contractors
- dispatches
- inbound SMS replies
- inbound email replies
- AI parsing
- quote response UI
- dashboard
- issue list
- notifications framework
- Twilio outbound/inbound
- Resend outbound/inbound

This document does NOT redesign the system.
It finishes the system.

==================================================
HIGH-LEVEL PRODUCT RULE
==================================================

Your product should not stop at:
- collecting quotes

It should continue through:
- choosing who to hire
- confirming the hire
- tracking the job
- preserving final history

This is what turns the product from:
- quote organizer

into:
- repair workflow system

==================================================
1. CORE USER FLOW
==================================================

STEP 1
Issue is dispatched to contractors

STEP 2
One or more contractors reply by SMS/email

STEP 3
Replies are parsed and shown on the issue page

STEP 4
User reviews responses and clicks:
SELECT CONTRACTOR

STEP 5
System creates a Job

STEP 6
Issue status changes from:
quotes_received → active_job

STEP 7
Selected contractor is shown everywhere relevant

STEP 8
Other quote options are visually closed

STEP 9
User tracks job until completion

STEP 10
Issue and job move into history

==================================================

This spec assumes you already have:
- Issue
- Contractor
- Dispatch

- Job

If Job already exists, use it.
If it does not fully include these fields, add what is missing.

--------------------------------------------------
2.1 ISSUE MODEL
--------------------------------------------------

Issue should support:

- id
- propertyId
- title
- description
- category
- urgency
- status
- selectedContractorId (nullable)
- selectedResponseId (nullable)
- createdAt
- updatedAt
- completedAt (nullable)

Status options should include:
- new
- classified
- awaiting_dispatch
- awaiting_quotes
- quotes_received
- active_job
- completed
- canceled
- archived

IMPORTANT:
If you currently use:
- contractor_selected
- scheduled
- in_progress

you may either:
A. keep them as internal job-specific states, or
B. simplify issue status to active_job and let job carry the stage

RECOMMENDATION:
Keep issue simpler:
- awaiting_quotes
- quotes_received
- active_job
- completed
- canceled
- archived

Let Job carry finer tracking state.

--------------------------------------------------
2.2 JOB MODEL
--------------------------------------------------

Job should include:

- id
- issueId
- contractorId
- selectedResponseId (nullable)
- status
- agreedPrice (nullable)
- estimatedAvailability (nullable)
- notes (nullable)
- scheduledFor (nullable)
- startedAt (nullable)
- completedAt (nullable)
- canceledAt (nullable)
- createdAt
- updatedAt

Recommended Job status values:
- selected
- scheduled
- in_progress
- completed
- canceled

--------------------------------------------------
2.3 CONTRACTOR RESPONSE MODEL
--------------------------------------------------

ContractorResponse should include or already contain:

- id
- issueId
- contractorId
- dispatchId
- rawMessage
- parsedPrice
- parsedAvailability
- parsedQuestionOrNotes
- sourceChannel (sms | email)
- receivedAt
- isSelected (boolean, default false)

If you do not already have `isSelected`, add it.
This simplifies UI.

--------------------------------------------------
2.4 DISPATCH MODEL
--------------------------------------------------

Dispatch should include:
- id
- issueId
- contractorId
- replyToken
- channel
- status
- providerMessageId
- createdAt
- updatedAt

Recommended dispatch statuses:
- draft
- sent

- replied

- failed

IMPORTANT:
Once contractor is chosen:
- selected contractor dispatch → accepted
- all other open dispatches → closed

==================================================
3. EXACT HIRING ACTION
==================================================

This is the missing first-class action.

--------------------------------------------------
3.1 WHERE IT APPEARS
--------------------------------------------------

On the Issue Detail page, inside each contractor response card.

Example card:

--------------------------------------------------
Test Plumber
$450
Available: Tomorrow
Question: What is the address and can I park there?

[Select Contractor]   [Reply]
--------------------------------------------------

--------------------------------------------------
3.2 USER CLICK FLOW
--------------------------------------------------

When user clicks:
[Select Contractor]

Show confirmation modal:

Title:
Hire Test Plumber?

Body:
This will mark Test Plumber as the selected contractor for this issue and start the job workflow.

Summary:
- Quote: $450
- Availability: Tomorrow

Buttons:
[Confirm Selection]
[Cancel]

--------------------------------------------------
3.3 CONFIRMATION RESULT
--------------------------------------------------

When user confirms:

System must do ALL of the following atomically if possible:

1. mark selected response:
- ContractorResponse.isSelected = true

2. update issue:
- selectedContractorId = contractorId
- selectedResponseId = responseId
- status = active_job

3. create job:
- issueId = issue.id
- contractorId = contractor.id
- selectedResponseId = response.id
- agreedPrice = parsedPrice
- estimatedAvailability = parsedAvailability
- status = selected

4. update dispatches:
- dispatch for selected contractor = accepted
- all other sent/replied dispatches = closed

5. create notification(s)
6. create timeline event(s)

IMPORTANT:
Do not allow partial state where contractor is selected but job does not exist.

==================================================
4. BACKEND API SPEC
==================================================

--------------------------------------------------
4.1 NEW ROUTE
--------------------------------------------------

Create route:

POST /api/issues/[id]/select-contractor

Body:
{
  "responseId": "..."
}

Authenticated user only.
User must own the issue/property.

--------------------------------------------------
4.2 VALIDATION RULES
--------------------------------------------------

Before selection:

1. confirm issue exists and belongs to user
2. confirm response exists
3. confirm response belongs to this issue
4. confirm issue is in state:
   - quotes_received
   OR
   - active_job only if you support re-selection (not recommended for V1)
5. confirm no completed/canceled job already exists for this issue in conflicting way
6. confirm contractor exists

--------------------------------------------------
4.3 V1 RULE: ONE ACTIVE SELECTION
--------------------------------------------------

For V1:
- issue can only have one selected contractor at a time
- once selected, changing contractor should require explicit “Replace contractor” flow later
- do NOT build replacement flow now unless you need it

If issue already has active job:
Return:
409 conflict
{
  "error": "CONTRACTOR_ALREADY_SELECTED"
}

==================================================
5. EXACT ISSUE STATUS TRANSITIONS
==================================================

This is the clean state model.

--------------------------------------------------
5.1 ISSUE STATUS FLOW
--------------------------------------------------

new
→ classified
→ awaiting_dispatch
→ awaiting_quotes
→ quotes_received
→ active_job
→ completed

Alternative exits:
→ canceled
→ archived

--------------------------------------------------
5.2 WHEN STATUS BECOMES quotes_received
--------------------------------------------------

As soon as first valid ContractorResponse is created.

This should happen automatically in inbound SMS/email processing.

Do NOT leave issue in awaiting_quotes after real reply exists.

--------------------------------------------------
5.3 WHEN STATUS BECOMES active_job
--------------------------------------------------

As soon as user selects contractor.

NOT when contractor replies.
NOT when contractor is contacted.
Only when user confirms selection.

--------------------------------------------------
5.4 WHEN STATUS BECOMES completed
--------------------------------------------------

When job status becomes completed.

==================================================
6. JOB LIFECYCLE (DETAILED)
==================================================

--------------------------------------------------
6.1 JOB STATE FLOW
--------------------------------------------------

selected
→ scheduled
→ in_progress
→ completed

Alternative:
selected / scheduled / in_progress
→ canceled

--------------------------------------------------
6.2 WHAT EACH STATE MEANS
--------------------------------------------------

selected
- contractor hired
- no explicit appointment locked yet

scheduled
- date/time agreed or expected

in_progress
- work has begun

completed
- work finished

canceled
- job canceled after selection

--------------------------------------------------
6.3 UI LABELS
--------------------------------------------------

Use user-facing labels:

selected → Contractor selected
scheduled → Scheduled
in_progress → In progress
completed → Completed
canceled → Canceled

14. HISTORY PAGE SPEC
==================================================

Current history page is too empty and too narrow.

==================================================
16. PAID PLAN EXPERIENCE RULES
==================================================

Because these are paid-user screens, every screen should reinforce value.

That means:
- visible responses
- visible progress
- visible decision points
- no dead empty states when real activity exists

If paid user receives a quote and sees:
No new notifications
and no clear hire action

that weakens perceived value immediately.
