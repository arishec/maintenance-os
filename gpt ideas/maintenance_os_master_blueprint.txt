MAINTENANCE OS — MASTER BUILD BLUEPRINT (FINAL)

==================================================
1. PRODUCT DEFINITION
==================================================

Maintenance OS is a workflow system for handling property maintenance:

Issue → Classify → Contact contractors → Collect replies → Compare → Select → Track → Store history

NOT a marketplace.
NOT a contractor directory.

Primary value:
- remove coordination friction
- centralize communication
- preserve repair history

==================================================
9. DISPATCH FLOW
==================================================

When user sends request:

1. create dispatch records
2. generate replyToken
3. include token in SMS/email
4. send via Twilio/Resend
5. update dispatch status

==================================================
10. REPLY CORRELATION (CRITICAL)
==================================================

Each dispatch has:
- replyToken

Inbound logic:

1. parse token from message
2. match dispatch by token
3. if no token:
   - find matching contractor + active dispatches
   - if ONE match → accept
   - if MULTIPLE → do NOT guess

Never match purely by "most recent".

==================================================
11. WEBHOOKS
==================================================

Twilio:
- inbound SMS
- status updates

Resend:
- inbound email

Rules:
- validate signatures
- handle retries safely
- ensure idempotency (no duplicate processing)

==================================================
12. AI USAGE
==================================================

Claude used for:

- issue classification
- urgency detection
- reply parsing

Rules:
- structured output only
- never guess missing data
- mark low confidence

==================================================
13. JOB SYSTEM
==================================================

One job per issue.

Job statuses:
- contractor_selected
- scheduled
- in_progress
- completed
- canceled

Rule:
Job status MUST sync to issue status.

==================================================
14. TENANT INTAKE
==================================================

Flow:

1. tenant submits issue via link
2. issue created immediately
3. classification attempted
4. failure does NOT block submission

Always return success if issue is created.

==================================================
15. CANCEL / ARCHIVE / DELETE
==================================================

Cancel:
- stops workflow
- remains in history

Archive:
- hides from active UI

Delete:
- NOT user-facing
- admin only (optional)

==================================================
16. COST TRACKING
==================================================

Track per issue:
- outbound SMS count
- inbound SMS count
- email count
- AI calls

Use upsert (never assume row exists).
