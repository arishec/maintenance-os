4. STRIPE WEBHOOK FALLBACK RESOLUTION
--------------------------------------------------

CURRENT ISSUE:
Stripe webhook handling may still rely too heavily on metadata.userId.

WHY THIS MATTERS:
If metadata is missing, malformed, or not present on a specific event:
- plan may not update
- user may remain free after payment
- cancellation may not downgrade correctly

WHAT TO DO:
In your Stripe webhook route, resolve the user in this order:

1. subscription metadata userId
2. stripeSubscriptionId
3. stripeCustomerId

RECOMMENDED LOGIC:
- try metadata.userId first
- if not found, query User where stripeSubscriptionId == subscription.id
- if still not found, query User where stripeCustomerId == subscription.customer

This should be implemented in a helper function used by:
- checkout.session.completed handling
- customer.subscription.updated
- customer.subscription.deleted

--------------------------------------------------

--------------------------------------------------

CURRENT ISSUE:
A Pro user may still be able to hit checkout route again and create a new session.

WHY THIS MATTERS:
- duplicate subscription flows
- billing confusion
- preventable support issue

WHAT TO DO:
In Stripe checkout route, before creating a session:

if (user.plan === 'pro') {
  return 400 / return redirect to billing portal later
}

MINIMUM BEHAVIOR:
Return:
{
  error: 'ALREADY_PRO',
  message: 'You already have an active Pro plan.'
}

--------------------------------------------------
6. WEBHOOK IDEMPOTENCY
--------------------------------------------------

CURRENT ISSUE:
This is one of the last important hardening items.
Twilio / Resend / Stripe can retry webhooks.

WHY THIS MATTERS:
Without idempotency:
- duplicate notifications
- duplicate ContractorResponse rows
- duplicate status transitions
- inflated metrics

WHAT TO DO:
Store provider IDs and short-circuit repeat processing.

FOR SMS:
- Twilio MessageSid

FOR EMAIL:
- provider inbound message ID if available

FOR STRIPE:
- event.id

RECOMMENDED RULE:
Before processing:
- check whether provider event/message ID has already been processed
- if yes, return success without reprocessing

7. ISSUE STATUS VS JOB STATUS DUPLICATION
--------------------------------------------------

CURRENT ISSUE:
There is still likely too much overlap between:
- Issue.status
and
- Job.status

Example duplicated concepts:
- contractor_selected
- scheduled
- in_progress

WHY THIS MATTERS:
Two sources of truth create drift.

RECOMMENDED MODEL:
Issue.status should stay simpler:
- awaiting_quotes
- quotes_received
- active_job
- completed
- canceled
- archived

Job.status should carry execution detail:
- selected
- scheduled
- in_progress
- completed
- canceled

WHAT TO DO:
Audit your current implementation and decide:

OPTION A:
Keep both, but enforce strict sync everywhere

OPTION B (recommended):
Simplify Issue.status and let Job.status hold detailed progression

If you do not want to refactor now:
At minimum document and enforce the sync rules clearly.
