1. app/dashboard/page.tsx
==================================================

LIKELY ISSUE

The current pattern is:

  let user;
  try {
    user = await requireDbUser();
  } catch {
    redirect('/sign-in');
  }

This often leaves TypeScript with a possibly-undefined `user`, even though `redirect()` never returns.
There is also unnecessary `as never` casting on Link hrefs.

WHAT TO CHANGE

A. Replace the try/catch with a helper pattern that returns a concrete user type.

Recommended:

```ts
import { redirect } from 'next/navigation';
import type { DbUser } from '@/lib/auth'; // only if you export a type; otherwise omit

async function getUserOrRedirect() {
  try {
    return await requireDbUser();
  } catch {
    redirect('/sign-in');
  }
}

export default async function DashboardPage() {
  const user = await getUserOrRedirect();

  // ...rest unchanged
}
```

If you do not want a helper:

```ts
const user = await requireDbUser().catch(() => redirect('/sign-in'));
```

That usually narrows correctly because redirect() is `never`.

B. Remove `as never` from dashboard stat links.

Current:
```ts
<Link key={stat.label} href={stat.href as never}>
```

Change to:
```ts
<Link key={stat.label} href={stat.href}>
```

C. Strongly type the stats array if needed.

```ts
const stats: Array<{ label: string; value: number; href: string }> = [
  { label: 'Open issues', value: openIssues, href: '/issues?view=open' },
  { label: 'Awaiting quotes', value: awaitingQuotes, href: '/issues?view=awaiting_quotes' },
  { label: 'Active jobs', value: activeJobs, href: '/issues?view=active_jobs' },
  { label: 'Properties', value: properties.length, href: '/properties' },
];
```

WHY

- removes possible undefined user state
- removes unnecessary route casting
- makes the file easier for TS to reason about

--------------------------------------------------
2C. TYPE THE PRISMA `where` CLAUSE PROPERLY
--------------------------------------------------

This is the biggest TypeScript improvement in the file.

Current:
```ts
const where: Record<string, unknown> = { ... }
```

That is too loose and usually causes trouble once you add `OR`, nested property filters, and relation filters.

Change the import:

```ts
import { Prisma } from '@prisma/client';
```

Then change the where object to:

```ts
const where: Prisma.IssueWhereInput = {
  propertyId: propertyFilter ? propertyFilter : { in: propertyIds },
  ...(statuses ? { status: { in: [...statuses] } } : {}),
  ...(urgencyFilter ? { urgency: urgencyFilter } : {}),
  ...(categoryFilter ? { category: categoryFilter } : {}),
};
```

Then the search block becomes valid:

```ts
if (searchFilter) {
  where.OR = [
    { title: { contains: searchFilter, mode: 'insensitive' } },
    { description: { contains: searchFilter, mode: 'insensitive' } },
    { property: { nickname: { contains: searchFilter, mode: 'insensitive' } } },
    { property: { addressLine1: { contains: searchFilter, mode: 'insensitive' } } },
  ];
}
```

NOTE:
If Prisma complains about one of the relation filters on `property`, rewrite them as:

```ts
if (searchFilter) {
  where.OR = [
    { title: { contains: searchFilter, mode: 'insensitive' } },
    { description: { contains: searchFilter, mode: 'insensitive' } },
    {
      property: {
        is: {
          nickname: { contains: searchFilter, mode: 'insensitive' },
        },
      },
    },
    {
      property: {
        is: {
          addressLine1: { contains: searchFilter, mode: 'insensitive' },
        },
      },
    },
  ];
}
```

Use whichever shape your generated Prisma client accepts.

--------------------------------------------------
2D. FIX THE URL BUILDER TYPE
--------------------------------------------------

Current:
```ts
return `/issues${qs ? `?${qs}` : ''}` as never;
```

Change to:
```ts
return `/issues${qs ? `?${qs}` : ''}`;
```

And type the function as:

```ts
function buildFilterUrl(
  base: Record<string, string | undefined>,
  overrides: Record<string, string | undefined>,
): string {
  const merged = { ...base, ...overrides };
  const params = new URLSearchParams();
  for (const [k, v] of Object.entries(merged)) {
    if (v) params.set(k, v);
  }
  const qs = params.toString();
  return `/issues${qs ? `?${qs}` : ''}`;
}
```

--------------------------------------------------
2E. REMOVE `as never` FROM LINKS
--------------------------------------------------

Anywhere in this file that does:

```ts
href={buildFilterUrl(...) as never}
```

Change to:
```ts
href={buildFilterUrl(...)}
```

WHY

- Prisma filters become type-safe
- relation search becomes much easier to validate
- TS no longer treats the where clause as a loose object bag
- route strings stop being force-cast

==================================================

CURRENT STATE

This file is very close to correct.

The only thing that typically breaks here is if `Button` does not support refs.
Your current `Button` now uses `forwardRef`, so that part is good.

WHAT TO CHECK

A. Confirm `Button` really exports a ref-forwarding component.

Your current button file already does this:

```ts
export const Button = React.forwardRef<HTMLButtonElement, ButtonProps>(...)
```

That is correct.

B. If this file still errors, the problem is usually prop compatibility on `AlertDialogAction` / `AlertDialogCancel`.

Safer version:

```ts
const AlertDialogAction = React.forwardRef<
  HTMLButtonElement,
  React.ComponentPropsWithoutRef<typeof Button>
>(({ className, ...props }, ref) => (
  <Button ref={ref} className={cn(className)} {...props} />
));
AlertDialogAction.displayName = 'AlertDialogAction';

const AlertDialogCancel = React.forwardRef<
  HTMLButtonElement,
  React.ComponentPropsWithoutRef<typeof Button>
>(({ className, ...props }, ref) => (
  <Button ref={ref} variant="outline" className={cn(className)} {...props} />
));
AlertDialogCancel.displayName = 'AlertDialogCancel';
```

WHY

Using `React.ComponentPropsWithoutRef<typeof Button>` is safer than re-declaring raw button HTML props if your Button component has variants and custom props.

5. lib/ai/client.ts
==================================================

CURRENT PROBLEM

The proxy export is convenient, but it is the weakest typed part of the AI client.

Current:
```ts
export const anthropic = new Proxy({} as Anthropic, {
  get(_, prop) {
    return (getAnthropicClient() as unknown as Record<string, unknown>)[prop as string];
  },
});
```

This can still confuse TS and gives poor autocomplete.

BEST FIX

Stop exporting the proxy.
Export only `getAnthropicClient()` and call it directly where needed.

STEP 1 — change lib/ai/client.ts to:

```ts
import Anthropic from '@anthropic-ai/sdk';

let _anthropic: Anthropic | null = null;

export function getAnthropicClient(): Anthropic {
  if (!_anthropic) {
    const key = process.env.ANTHROPIC_API_KEY;
    if (!key || key === 'REPLACE_ME' || key === 'sk-ant-REPLACE_ME') {
      throw new Error('Anthropic API key not configured. Set ANTHROPIC_API_KEY in .env');
    }
    _anthropic = new Anthropic({ apiKey: key });
  }
  return _anthropic;
}
```

STEP 2 — update classify-issue.ts

Change:
```ts
import { anthropic } from '@/lib/ai/client';
```

To:
```ts
import { getAnthropicClient } from '@/lib/ai/client';
```

Then before the API call:
```ts
const anthropic = getAnthropicClient();
```

STEP 3 — update parse-contractor-reply.ts the same way.

WHY

- removes the proxy hack entirely
- improves autocomplete and inference
- avoids cast-heavy code
- easier to debug

If you want to keep the proxy for backward compatibility, you can, but the direct getter approach is the clean fix.

6. lib/timeline.ts
==================================================

LIKELY ISSUE

The payload type is too loose:

```ts
payload?: Record<string, unknown>;
```

Then you serialize it into JSON for Prisma:

```ts
eventPayloadJson: input.payload ? JSON.parse(JSON.stringify(input.payload)) : undefined
```

This works at runtime, but Prisma usually wants a JSON input type rather than a generic record.

FIX

Import Prisma:

```ts
import { prisma } from '@/lib/prisma';
import type { Prisma } from '@prisma/client';
```

Then change the interface to:

```ts
type ActorType = 'user' | 'tenant' | 'system' | 'contractor';

interface LogTimelineEventInput {
  propertyId: string;
  issueId?: string;
  jobId?: string;
  actorType: ActorType;
  actorLabel?: string;
  eventType: string;
  payload?: Prisma.InputJsonValue;
}
```

Then simplify the create call:

```ts
export async function logTimelineEvent(input: LogTimelineEventInput) {
  return prisma.timelineEvent.create({
    data: {
      propertyId: input.propertyId,
      issueId: input.issueId,
      jobId: input.jobId,
      actorType: input.actorType,
      actorLabel: input.actorLabel,
      eventType: input.eventType,
      eventPayloadJson: input.payload,
    },
  });
}
```

WHY

- Prisma gets the JSON type it expects
- no stringify/parse hack needed
- cleaner and safer
