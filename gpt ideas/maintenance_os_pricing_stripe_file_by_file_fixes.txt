6. app/api/stripe/checkout/route.ts
==================================================

CURRENT STATUS

This route is directionally correct:
- create/reuse Stripe customer
- create checkout session
- attach metadata
- return checkout URL

--------------------------------------------------
REAL FIX #1 — GUARD AGAINST ALREADY-PRO USERS
--------------------------------------------------

PROBLEM

If the user is already Pro, the route may still create a new checkout session.

That is sloppy and could cause duplicate billing flows.

--------------------------------------------------
FIX
--------------------------------------------------

Before creating checkout session, add:

```ts
if (user.plan === 'pro') {
  return NextResponse.json(
    { error: 'ALREADY_PRO', message: 'You already have an active Pro plan.' },
    { status: 400 }
  );
}
```

If you later add a billing portal, this would instead redirect them there.

--------------------------------------------------
REAL FIX #2 — KEEP URL ENV CLEAN
--------------------------------------------------

If you currently use something like:

```ts
success_url: `${process.env.NEXT_PUBLIC_APP_URL}/dashboard?upgraded=true`,
cancel_url: `${process.env.NEXT_PUBLIC_APP_URL}/pricing`,
```

That is acceptable only if the env is guaranteed correct.

--------------------------------------------------
RECOMMENDED IMPROVEMENT
--------------------------------------------------

Use one consistent environment variable across the whole app, for example:

```env
NEXT_PUBLIC_SITE_URL=https://yourdomain.com
```

Then in code:

```ts
const siteUrl = process.env.NEXT_PUBLIC_SITE_URL!;
```

and:

```ts
success_url: `${siteUrl}/dashboard?upgraded=true`,
cancel_url: `${siteUrl}/pricing`,
```

This avoids domain drift between:
- SEO
- checkout
- webhooks
- metadata

--------------------------------------------------
REAL FIX #3 — STORE CUSTOMER ID DEFENSIVELY
--------------------------------------------------

If customer is created/reused, ensure it is written back to the user:

```ts
await prisma.user.update({
  where: { id: user.id },
  data: {
    stripeCustomerId: customer.id,
  },
});
```

Only do this if missing or changed.

--------------------------------------------------
RECOMMENDED SHAPE
--------------------------------------------------

Pseudo-structure:

```ts
const user = await requireDbUser();

if (user.plan === 'pro') {
  return NextResponse.json(
    { error: 'ALREADY_PRO', message: 'You already have an active Pro plan.' },
    { status: 400 }
  );
}

const siteUrl = process.env.NEXT_PUBLIC_SITE_URL!;
const priceId = process.env.STRIPE_PRO_PRICE_ID!;
const stripe = getStripe();

let customerId = user.stripeCustomerId;

if (!customerId) {
  const customer = await stripe.customers.create({
    email: user.email,
    metadata: { userId: user.id },
  });

  customerId = customer.id;

  await prisma.user.update({
    where: { id: user.id },
    data: { stripeCustomerId: customerId },
  });
}

const session = await stripe.checkout.sessions.create({
  mode: 'subscription',
  customer: customerId,
  line_items: [{ price: priceId, quantity: 1 }],
  success_url: `${siteUrl}/dashboard?upgraded=true`,
  cancel_url: `${siteUrl}/pricing`,
  metadata: {
    userId: user.id,
  },
  subscription_data: {
    metadata: {
      userId: user.id,
    },
  },
});

return NextResponse.json({ url: session.url });
```

--------------------------------------------------
IMPORTANT
--------------------------------------------------

Putting `userId` in BOTH:
- checkout session metadata
- subscription metadata

is safer than only one layer.

7. app/api/stripe/webhook/route.ts
==================================================

CURRENT STATUS

This is generally good:
- signature verification exists
- it handles subscription lifecycle events

But it is still too dependent on subscription metadata.

--------------------------------------------------
PROBLEM
--------------------------------------------------

If you only do:

```ts
const userId = subscription.metadata?.userId;
```

then a malformed or missing metadata case can break plan syncing.

--------------------------------------------------
FIX
--------------------------------------------------

Use a fallback resolution strategy.

ORDER OF RESOLUTION:
1. Try `subscription.metadata.userId`
2. If missing, find user by `stripeSubscriptionId`
3. If still missing, find user by `stripeCustomerId`

--------------------------------------------------
RECOMMENDED HELPER
--------------------------------------------------

Add a helper function in the webhook file or shared billing helper:

```ts
async function resolveUserForSubscription(subscription: Stripe.Subscription) {
  const metadataUserId = subscription.metadata?.userId;

  if (metadataUserId) {
    const user = await prisma.user.findUnique({
      where: { id: metadataUserId },
    });
    if (user) return user;
  }

  const bySubscription = await prisma.user.findFirst({
    where: { stripeSubscriptionId: subscription.id },
  });
  if (bySubscription) return bySubscription;

  const customerId =
    typeof subscription.customer === 'string'
      ? subscription.customer
      : subscription.customer?.id;

  if (customerId) {
    const byCustomer = await prisma.user.findFirst({
      where: { stripeCustomerId: customerId },
    });
    if (byCustomer) return byCustomer;
  }

  return null;
}
```

--------------------------------------------------
THEN USE IT IN EACH EVENT
--------------------------------------------------

For `checkout.session.completed`:
- extract subscription ID if present
- store `stripeSubscriptionId`
- set `plan = 'pro'`

For `customer.subscription.updated`:
- resolve user using helper
- if active/trialing/etc. set plan = 'pro'
- if canceled/unpaid/incomplete_expired etc. decide downgrade behavior

For `customer.subscription.deleted`:
- resolve user using helper
- set:
  - `plan = 'free'`
  - `stripeSubscriptionId = null`

--------------------------------------------------
RECOMMENDED STATUS MAPPING
--------------------------------------------------

Treat these as active paid access:
- active
- trialing

Treat these as not paid:
- canceled
- unpaid
- incomplete_expired

Be careful with:
- past_due
- incomplete

You need to decide whether those should keep access briefly or not.

For a simple V1:
- active / trialing => pro
- everything else => free

8. lib/stripe.ts (or equivalent Stripe client file)
==================================================

RECOMMENDED SHARED CLIENT
==================================================

Create a single Stripe client helper if you do not already have one.

File:
`lib/stripe.ts`

```ts
import Stripe from 'stripe';

let _stripe: Stripe | null = null;

export function getStripe() {
  if (!_stripe) {
    const apiKey = process.env.STRIPE_SECRET_KEY;
    if (!apiKey) {
      throw new Error('Missing STRIPE_SECRET_KEY');
    }

    _stripe = new Stripe(apiKey, {
      apiVersion: '2024-06-20',
    });
  }

  return _stripe;
}
```

Use this everywhere instead of creating new Stripe instances ad hoc.

WHY:
- cleaner typing
- shared config
- easier maintenance
