==================================================
1. DOCUMENT PURPOSE
==================================================

This document is intended to be handed directly to a developer or small product engineering team.

The goal is to remove ambiguity and lock the initial product direction so the team can begin implementation without reopening core product decisions.

This version explicitly resolves the prior open stack questions and adds a cost / unit economics section because outbound and inbound messaging costs are part of the product model.

==================================================
2. PRODUCT SUMMARY
==================================================

Working product name:
Maintenance OS

One-line product definition:
A responsive web application for homeowners and landlords that converts a maintenance issue into a structured workflow: report issue -> classify issue -> contact contractors -> compare responses -> select contractor -> track job -> retain property repair history.

Core thesis:
The product is not a marketplace. It is a maintenance coordination and decision workflow engine.

Primary outcome for the user:
Reduce the time, stress, and fragmentation involved in getting repair problems handled.

Primary value for homeowners:
“Help me get this fixed without the headache.”

Primary value for landlords:
“Help me coordinate maintenance across properties without chasing contractors.”

What the product must do in V1:
- collect maintenance issues
- classify issue type and urgency
- contact contractors through SMS and/or email
- collect and structure contractor responses
- help the user compare and choose
- track the job through completion
- keep history tied to each property

What the product must not become in V1:
- contractor marketplace
- rent collection tool
- full property management suite
- live AI calling agent
- autonomous contractor negotiator
- payment processor
- lease administration system

==================================================
5. CORE PRODUCT PRINCIPLES
==================================================

1. Property-centric system
The model revolves around properties, not user type.
A homeowner typically has one property.
A landlord has multiple properties.
Same engine.

2. Same backend, different surface area
Homeowner view should feel guided and simple.
Landlord view should feel operational and scalable.

3. AI is assistive, not theatrical
AI should classify, extract, summarize, and structure.
AI should not pretend to negotiate, diagnose with certainty, or fully automate judgment.

4. Contractors should have near-zero friction
No contractor portal required in V1.
No contractor signup required in V1.
SMS and email first.

5. Human approval is required for key decisions
The user explicitly chooses:
- which contractors to contact
- whether extracted data is correct
- which contractor to select
- when a job is completed

6. Raw data is always preserved
Always store:
- original issue description
- original uploaded photos
- original outbound message
- original inbound contractor message

Structured AI output is secondary to the raw record.

==================================================
8. AI DECISION: WHY CLAUDE IS PRIMARY
==================================================

The hardest AI problem in this product is not generic classification.
It is messy real-world language extraction.

Examples of contractor replies the system must handle:
- “can probably do tomorrow afternoon around 250 if it’s straightforward”
- “I can come Friday. Need to see it first. Usually 175-300”
- “yes”
- “call me”
- “depends on parts”
- “earliest is Tues after 3”
- “could be 200 labor plus materials”

These are short, ambiguous, incomplete, and operationally messy.
This favors an AI model that is strong at interpreting real-world text while still producing schema-constrained outputs.

Claude should be used for:
- issue classification
- urgency detection
- issue summary generation
- contractor reply parsing
- “needs review” decisioning when confidence is low

Implementation rule:
Even though Claude is primary, the code should still isolate the AI provider behind an internal service layer so future provider changes do not require broad application rewrites.

==================================================

- classify issue category

- propose issue title
- summarize issue

- place live calls

- urgency
- reasoning_summary

- recommended_trade

Category enum:
- plumbing
- electrical
- HVAC
- roofing
- appliance
- structural
- pest
- cleaning
- exterior
- general_handyman
- unknown

Urgency enum:
- emergency
- high
- medium
- low

Suggested timeframe values:
- immediately
- today
- within_24_hours
- within_2_to_3_days
- within_1_week

14.4 Contractor reply extraction output schema
Fields:
- availability_text
- availability_date nullable
- estimate_low nullable
- estimate_high nullable
- flat_estimate nullable
- notes
- follow_up_question nullable
- confidence_score
- requires_review boolean

14.5 Parsing rules
- Always store raw reply first
- Run extraction second
- If confidence is low, flag requires_review
- If no useful structured info can be extracted, preserve raw reply and show waiting for manual interpretation

14.6 Claude prompt design guidance
Use schema-constrained responses wherever possible.
Prompts should:
- include allowed category enums
- include known issue category and contractor trade when parsing replies
- instruct model to prefer null over guessing
- avoid fabricated dates or estimates
- explicitly mark ambiguous content as requires_review

16. UNIT ECONOMICS AND COST TRACKING
==================================================

This section is critical and should be implemented as a first-class product and engineering concern.

16.1 Why this matters
This product has variable cost drivers per issue:
- outbound SMS
- inbound SMS
- outbound email
- AI classification call
- AI reply parsing calls
- image storage
- file delivery bandwidth
- reminder / follow-up messages

If not tracked early, the product can look useful but develop weak gross margins.

16.2 Per-issue cost model to track
For every issue, the system should eventually be able to estimate:
- number of contractors contacted
- number of outbound SMS messages sent
- number of inbound SMS messages received
- number of emails sent
- number of AI calls used
- number and size of images stored
- estimated total communication and AI cost per issue

16.3 Practical product implications
The product should not assume “contact as many contractors as possible.”
Contacting 2 contractors may be much better economically than contacting 5 if response rates are high enough.

This means the product should later track:
- average contractors contacted per issue
- average response rate
- median time to first reply
- cost per resolved issue
- cost per quote received

16.4 Recommended V1 analytics fields
At minimum, store:
- dispatch count per issue
- SMS count per issue
- reply count per issue
- AI parse count per issue

Even if cost is not shown in UI, it should be measurable internally.

16.5 Why Twilio cost needs special attention
This product is SMS-heavy by design.
Example issue flow could involve:
- 3 outbound requests
- 2 inbound replies
- 1 follow-up reminder
- 1 user notification SMS

That can scale quickly as usage grows.
Engineering should avoid hidden message multiplication and should be intentional about reminders and status alerts.

16.6 Recommendation
Instrument communication events from day one.
Do not wait until billing pain forces retroactive measurement.

19. BACKEND LOGIC REQUIREMENTS
==================================================

19.1 User bootstrap logic
On first authenticated session:
- read Clerk user
- upsert local users row
- store clerk_user_id
- keep email in sync if needed

19.2 Issue creation logic
On issue submission:
- validate property ownership or intake token validity
- create issue record with status = new
- upload photo(s) to Supabase Storage
- create issue_photo rows
- queue classification job
- append timeline event

19.3 Classification logic
- call Claude through internal AI service
- validate structured response
- update issue with classification fields
- if low confidence, category may remain unknown and require user review
- set issue status to classified or awaiting_dispatch depending on UX path
- append timeline event

19.4 Dispatch logic
When user selects contractors:
- create dispatch rows
- generate outbound message for each recipient
- send via Twilio SMS and/or email provider
- store provider identifiers
- update dispatch statuses from callbacks
- increment issue usage metrics
- append timeline event

19.5 Inbound contractor reply logic
On incoming SMS/email:
- resolve provider payload to dispatch
- store raw inbound message
- run Claude extraction
- create contractor_response row
- update dispatch status to replied
- notify user
- increment usage metrics
- append timeline event

19.6 Contractor selection logic
When user chooses contractor:
- create job if absent
- store selected response
- update issue status
- optionally mark other dispatches as declined or leave passive
- append timeline event

19.7 Job progression logic
Mark scheduled:
- update job status
- update issue status
- store scheduled_for if provided

Mark in progress:
- update job status
- update issue status

Mark complete:
- update job status to completed
- update issue status to completed
- set completed timestamps
- append timeline event

19.8 Manual override logic
The user must be able to:
- edit category
- edit urgency
- edit extracted estimate
- enter an offline quote manually
- mark a phone-call response manually

20. CLAUDE INTEGRATION REQUIREMENTS
==================================================

20.1 Internal AI service layer
Create a dedicated internal AI module, for example:
- /lib/ai/classifyIssue.ts
- /lib/ai/parseContractorReply.ts
- /lib/ai/client.ts

This service layer should:
- handle model selection
- enforce structured outputs
- centralize prompt templates
- centralize retries
- centralize cost and token logging if implemented

20.2 Issue classification contract
Input:
- issue description
- property location field
- safety/water/heat/power signals
- image summaries or direct vision input if enabled
- allowed enum list

Output:
- title
- category
- urgency
- reasoning_summary
- suggested_timeframe
- recommended_trade
- confidence_score

20.3 Contractor reply parse contract
Input:
- raw reply text
- issue category
- contractor trade
- current timestamp context

Output:
- availability_text
- availability_date
- estimate_low
- estimate_high
- flat_estimate
- notes
- follow_up_question
- confidence_score
- requires_review

20.4 Prompting rules
- prefer null instead of guessing
- never invent an estimate if one is not present
- never convert vague language into precise dates unless clearly stated
- preserve uncertainty
- mark low confidence clearly

==================================================
24. SECURITY AND DATA RULES
==================================================

Requirements:
- all authenticated routes must map Clerk identity to local user
- enforce ownership checks at property, issue, contractor, and job level
- intake links must be revocable and tokenized
- uploaded photos must not be publicly exposed without controlled URLs
- webhook endpoints must verify provider signatures where supported
- avoid leaking contractor contact data across account boundaries

Auditability requirement:
Timeline events should preserve who did what and when.

==================================================
25. EDGE CASES AND FAILURE HANDLING
==================================================

System must handle:
- contractor replies “yes”
- contractor replies “call me”
- contractor replies with no price
- contractor provides only a date
- user gets quote by phone and enters manually
- AI misclassifies issue
- duplicate issue submissions
- duplicate webhook deliveries
- contractor message fails to deliver
- issue has no photo
- emergency issue where user should not wait for quote comparison

Low-information reply handling:
If reply is not parseable into a clean estimate/availability pair:
- keep raw message
- mark needs review
- allow manual interpretation

==================================================
26. EMERGENCY UX RULES
==================================================

If urgency is emergency:
- show prominent warning banner
- tell user to take immediate safety action when appropriate
- do not force a slow quote-comparison flow before action
- surface “call now” path if a saved emergency contractor exists

Examples:
- gas smell
- exposed sparking wires
- severe flooding
- no heat in dangerous weather conditions

31. FINAL SUMMARY FOR THE DEVELOPER
==================================================

Build a responsive web application called Maintenance OS.

The app must support:
- homeowners with one or more properties
- landlords with multiple properties
- tenant issue intake through secure property links
- contractor communication through SMS/email without contractor login

The core lifecycle is:
1. issue reported
2. AI classifies issue
3. user selects contractors
4. system sends requests
5. contractor replies are captured and parsed
6. user compares responses
7. user selects contractor
8. job is tracked through completion
9. property history is preserved

The app should feel simple for homeowners and operational for landlords.

Do not build a marketplace.
Do not require contractor accounts.
Do not over-automate decisions.
Do track communication costs from the beginning.
Do make Claude parsing and review workflows robust.

End of Version 1.1 specification.
