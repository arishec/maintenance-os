========================================
1. PRODUCT SUMMARY
========================================

Product name (working):
Maintenance OS

One-line description:
A maintenance coordination platform for homeowners and landlords that turns a reported issue into a classified job, sends quote requests to contractors, aggregates responses, and helps the user choose and track the repair.

Core product thesis:
The product should NOT be a contractor marketplace. It should be a workflow engine that helps users coordinate repairs faster with less texting, calling, and follow-up.

Primary value:
- homeowner: “Help me get this fixed without the headache.”
- landlord: “Help me coordinate maintenance across one or more properties without chasing contractors.”

What the product is:
- issue intake
- issue classification
- urgency detection
- contractor dispatch
- response aggregation
- quote comparison
- scheduling / job tracking
- history by property

What the product is not in V1:
- full property management software
- rent collection software
- contractor marketplace
- contractor bidding exchange
- live AI voice calling
- automatic negotiation
- payments / escrow
- insurance claims platform

========================================
2. TARGET USERS
========================================

2.1 Primary user types

A. Homeowner
- typically manages 1 property
- reports issues themselves
- may or may not already have contractors
- needs quick help finding or contacting someone
- wants minimal complexity

B. Landlord / small property owner
- manages multiple properties
- may have tenants reporting issues
- wants one place to see open issues across properties
- may already have preferred contractors
- needs fast coordination and status tracking

2.2 Secondary users

C. Tenant
- can report an issue to the property owner through a shared intake link
- does not need a full account in V1
- may receive updates, but does not manage contractors

D. Contractor
- receives SMS/email job requests
- can respond without logging in
- does not need a full dashboard in V1

========================================
3. PRODUCT POSITIONING
========================================

Core positioning:
“Stop texting five contractors. Send one request. Get all responses in one place.”

For homeowners:
- report issue
- get help fast
- compare options clearly
- keep repair history organized

For landlords:
- manage maintenance across properties
- centralize tenant requests
- route jobs to preferred vendors
- keep a documented record of repairs

Critical positioning rule:
Do not market V1 as “find the best plumber near you.”
Do not market V1 as a marketplace.
Contractors are part of the workflow, not the product itself.

========================================
4. V1 PRODUCT SCOPE
========================================

The MVP must support both homeowners and landlords using the same core backend.

4.1 Supported workflows in V1
- create account
- add one or more properties
- submit issue with text + optional photo(s)
- AI classifies issue and urgency
- user chooses contractors to contact
- platform sends SMS/email requests
- platform stores inbound replies
- AI extracts quote / availability / notes from replies
- platform shows comparison table
- user marks contractor as selected
- job moves through lifecycle
- property maintains issue history

4.2 Out of scope for V1
- contractor app
- contractor portal
- auto scheduling on contractor calendars
- payments
- background checks / licensing verification
- AI phone calls
- recurring maintenance workflows
- maintenance subscription plans
- insurance carrier integrations
- lease management
- work order accounting integrations

========================================
5. KEY PRODUCT PRINCIPLES
========================================

1. Property-centric model
Everything revolves around properties, not user type.
A homeowner has one property.
A landlord has multiple properties.
Same backend.

2. Same core engine, different surface area
Homeowner view = simple and guided.
Landlord view = multi-property dashboard and filters.

3. AI should structure, not pretend
AI classifies, extracts, summarizes, and recommends.
AI should not impersonate human authority or negotiate automatically.

4. Minimize contractor friction
Contractors should not need accounts in V1.
SMS/email first.

5. Human decision remains explicit
The user always approves:
- which contractors to contact
- which quote to accept
- whether a job is completed

========================================
8. USER FLOWS
========================================

8.1 Homeowner main flow
1. Sign up
2. Add property
3. Click “Report issue”
4. Enter issue description and upload photo(s)
5. AI classifies category + urgency
6. User reviews classification
7. User selects saved contractor(s) OR adds a contractor
8. Platform sends quote request SMS/email
9. Replies arrive and are parsed
10. User sees comparison table
11. User selects contractor
12. Job status progresses:
   new -> awaiting_quotes -> quotes_received -> contractor_selected -> scheduled -> in_progress -> completed
13. Repair history stored

8.2 Landlord main flow
1. Sign up
2. Add multiple properties
3. Share tenant issue intake link per property
4. Tenant submits issue
5. AI classifies category + urgency
6. Landlord reviews open issue queue
7. Landlord dispatches to saved contractor(s)
8. Replies are parsed and displayed
9. Landlord selects contractor
10. Job status tracked and updates visible in dashboard

8.3 “Add contractor” flow
V1:
- manual entry:
  - contractor name
  - phone
  - email
  - trade
  - notes

V1.1 or V2:
- search nearby businesses using Google Places / similar provider
- user selects one and saves it
Important:
This is not phrased as “recommended by us.”
This is phrased as “add a local business to contact.”

12. AI BEHAVIOR SPEC
========================================

12.1 AI use cases in V1
- classify issue category
- determine urgency
- summarize issue
- extract structured fields from contractor replies
- optionally suggest best next step

12.2 AI should not do in V1
- directly negotiate price
- autonomously select contractor
- automatically schedule without user approval
- call vendors
- guarantee technical correctness of repair advice

12.3 Classification output schema
Input:
- free text
- optional photo(s)
- location tag
- safety/water/heat/power toggle

Output fields:
- title
- category
- subcategory (optional)
- urgency
- reasoning_summary
- suggested_timeframe
- confidence_score
- recommended_trade

Category enum examples:
- plumbing
- electrical
- HVAC
- roofing
- appliance
- structural
- pest
- cleaning
- exterior
- general_handyman
- unknown

Urgency enum:
- emergency
- high
- medium
- low

Suggested timeframe examples:
- immediately
- today
- within_24_hours
- within_2_to_3_days
- within_1_week

12.4 Contractor reply extraction schema
Input:
- raw SMS or email response text
Output:
- availability_text
- availability_date (nullable)
- estimate_low (nullable)
- estimate_high (nullable)
- flat_estimate (nullable)
- notes
- follow_up_question (nullable)
- confidence_score

Important:
Always store raw reply.
AI extraction is an assistive layer, not source of truth.

========================================
16. BACKEND LOGIC REQUIREMENTS
========================================

16.1 Issue creation logic
On issue creation:
- validate property ownership / link validity
- upload photos
- create issue record with temporary status = new
- queue AI classification job
- update issue to classified + awaiting_dispatch after classification

16.2 Classification logic
- call LLM with strict structured output
- if model confidence too low, set category = unknown and flag manual review
- save reasoning summary
- do not auto-dispatch

16.3 Dispatch logic
When user selects contractors:
- create dispatch rows
- build outbound messages
- send through SMS and/or email provider
- update dispatch status based on provider callbacks
- append timeline event

16.4 Inbound response logic
On inbound SMS/email:
- identify contractor + dispatch
- create contractor_response with raw content
- run extraction parser / LLM
- update dispatch status = replied
- notify user
- append timeline event

16.5 Contractor selection logic
When user chooses one response:
- create or update job
- update issue status = contractor_selected
- mark selected dispatch/response
- append timeline event

16.6 Completion logic
When completed:
- update job.status = completed
- update issue.status = completed
- set completed_at timestamps
- append timeline event

17.2 Issue classification prompt requirements
Input bundle:
- issue description

- safety/water/heat/power toggle
- image summary if vision used
- allowed categories enum
- allowed urgency enum

Must return:
- title
- category
- urgency
- reasoning summary
- suggested timeframe
- recommended trade
- confidence

17.3 Contractor response extraction prompt requirements
Input:
- raw message
- known issue category
- known contractor trade
- current date/time

Must return:
- whether response contains estimate
- estimate values
- availability
- notes
- if human review is needed

========================================
19. MOBILE VS WEB
========================================

Recommendation for V1:
Build as a responsive web app, not a native mobile app.

Reason:
- faster to ship
- easier for homeowner and landlord use
- issue reporting and dashboard both work well in responsive web
- contractor side does not need an app

Mobile requirements:
- issue reporting must be excellent on mobile
- photo upload must be easy on mobile
- dashboard must remain usable on tablet/phone

Desktop requirements:
- landlord table views should be optimized for desktop

Future:
A native mobile wrapper can be considered later if usage proves strong.

========================================
23. NON-FUNCTIONAL REQUIREMENTS
========================================

Performance:
- dashboard load under 2s for normal accounts
- issue creation should feel instant; classification can be async with loading state

Reliability:
- message sending must be idempotent
- webhook handlers must be retry-safe
- raw inbound messages must always be stored

Security:
- secure auth
- property ownership checks on all endpoints
- tenant intake links must be tokenized and revocable
- image uploads must be access-controlled
- PII handled carefully

Auditability:
- timeline events should preserve who did what and when

========================================
24. EDGE CASES
========================================

Must handle:
- contractor replies with only “yes”
- contractor replies by phone instead of SMS/email
- user receives quote offline and wants to enter it manually
- AI misclassifies category
- multiple contractors reply with vague estimates
- duplicate dispatch sends
- tenant submits duplicate issue
- issue has no photo
- issue is true emergency and system should suggest immediate action

Emergency handling rule:
If issue appears emergency-level (e.g. gas smell, electrical sparking, major flooding), UI should clearly tell user to take immediate safety action and not rely solely on quote collection.

========================================
25. EMERGENCY UX RULES
========================================

If urgency = emergency:
- show red warning banner
- show “Contact emergency services or your emergency vendor immediately if there is danger.”
- allow dispatch, but do not make the user wait through a slow quote-comparison path
- include a “call now” style action if saved emergency contractor exists

28. WHAT “DONE” LOOKS LIKE FOR V1
========================================

A homeowner can:
- create an account
- add their home
- report a leak with a photo
- get classification
- contact 2 contractors they added
- receive replies
- compare those replies
- pick one
- track the repair
- later see the full history

A landlord can:
- create an account
- add 3 rental properties
- share a tenant intake link for one property
- receive a tenant issue submission
- dispatch that issue to saved contractors
- compare responses
- select one
- track completion
- see all open issues across properties

If those two flows work smoothly, the MVP is successful.

========================================
29. CRITICAL PRODUCT DECISIONS TO PRESERVE
========================================

1. Do not turn V1 into a contractor marketplace.
2. Do not require contractor signup.
3. Build responsive web first, not native app first.
4. Keep the data model property-centric.
5. Keep AI assistive and reviewable.
6. Always store raw contractor replies.
7. Support both homeowners and landlords through the same core engine.

========================================
31. FINAL PRODUCT SUMMARY FOR THE DEVELOPER
========================================

Build a responsive web application called Maintenance OS.

The app should support:
- homeowners with one property
- landlords with multiple properties

The core workflow is:
1. issue reported
2. AI classifies issue
3. user selects contractors
4. platform sends requests by SMS/email
5. replies are parsed and displayed
6. user chooses contractor
7. job is tracked to completion
8. issue history is retained per property

Do not build a marketplace.
Do not require contractor accounts.
Do not over-automate decisions.
Optimize for fast issue coordination and clear operational visibility.

End of specification.
