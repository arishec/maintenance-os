11. RESEND-DISPATCH ROUTE OVERWRITES EXISTING replyToken
--------------------------------------------------

File:
app/api/issues/[id]/resend-dispatch/route.ts

Current behavior:
- generates a new replyToken
- updates the existing dispatch row with the new token

Why this is dangerous:
If a contractor replies to the ORIGINAL email or SMS later,
that older token may no longer match the dispatch.

That means:
- late reply can be lost
- lookup becomes historically inconsistent
- original email thread is effectively invalidated

This is a real logic issue.

--------------------------------------------------
BETTER OPTIONS
--------------------------------------------------

Option A (recommended):
Create a NEW dispatch row for the resend.
Treat resend as a new dispatch attempt.

That preserves:
- original token
- original provider message ID
- full history of attempts

Option B:
Keep a token history model / old token table.
This is more complexity and less clean than Option A.

--------------------------------------------------
RECOMMENDED FIX
--------------------------------------------------

Do NOT mutate the original replyToken on resend.
Create a new Dispatch row with:
- same issueId
- same contractorId
- same channel
- new replyToken
- new outboundMessage/providerMessageId
- status = sent

Then timeline can log “dispatch_resent” while data stays trustworthy.

--------------------------------------------------
12. reply-to-contractor ROUTE IS TOO LOOSE IN DISPATCH CHOICE
--------------------------------------------------

File:
app/api/issues/[id]/reply-to-contractor/route.ts

Current behavior:
- if contractorId provided, use matching dispatch
- otherwise fallback to first active dispatch

Why this is risky:
If future UI ever fails to send contractorId or contractorResponseId correctly,
backend may reply to the wrong contractor.

Even if the UI currently behaves, the API should be stricter.

--------------------------------------------------
FIX
--------------------------------------------------

Require one of:
- contractorResponseId
OR
- contractorId

Best:
Make contractorResponseId required for this route.

Then derive contractor/dispatch context from the response.
This guarantees you are replying in context of a known contractor question/quote.

If you keep contractorId optional:
- validate more strictly
- do not silently fall back to issue.dispatches[0] unless you are absolutely sure

Recommended:
Remove the “first dispatch” fallback.
