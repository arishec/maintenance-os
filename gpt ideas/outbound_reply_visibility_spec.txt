MAINTENANCE OS
OUTBOUND REPLY VISIBILITY SPEC
(HOW TO SHOW THAT THE USER REPLIED TO A CONTRACTOR, AND WHAT THEY SAID)

==================================================
PURPOSE
==================================================

This document defines exactly how to fix the current product gap:

Current problem:
- contractor asks a follow-up question
- user replies from inside the app
- message is sent
- BUT the issue page does not clearly show:
  - that the reply was sent
  - what the reply said
  - when it was sent
  - which channel was used

This creates a trust problem.

The app must become the system of record for the repair conversation.

==================================================
GOAL
==================================================

After a user replies to a contractor from inside the app, the issue detail page
must clearly show:

1. the contractor’s question
2. the user’s reply
3. the channel used (email or SMS)
4. when the reply was sent
5. whether sending succeeded or failed

This should be implemented cleanly, without building a full chat system yet.

==================================================
PRODUCT DECISION
==================================================

For V1:
DO NOT build a full threaded messaging system.

Instead:
- store outbound messages
- show them directly under the relevant contractor response card

This is simpler, cleaner, and enough for the product right now.

==================================================
1. WHAT THE USER SHOULD SEE
==================================================

On the issue detail page, inside the contractor quote card, after the contractor
question block, show:

--------------------------------------------------
Contractor question:
What is the address and where do I park?

Your reply:
Park in the street.

Sent via Email · Mar 29, 7:21 AM
--------------------------------------------------

If multiple outbound replies exist later, show the most recent one for V1.
Optionally add:
"View all messages" later.

==================================================
2. DATABASE / DATA MODEL
==================================================

You need to store outbound replies as first-class records.

--------------------------------------------------
OPTION A (RECOMMENDED)
--------------------------------------------------

Create a new model:

ContractorMessage

Fields:
- id
- issueId
- contractorId
- contractorResponseId (nullable)
- direction ("inbound" | "outbound")
- channel ("sms" | "email")
- messageBody
- providerMessageId (nullable)
- sendStatus ("sent" | "failed")
- createdAt
- updatedAt

--------------------------------------------------
RECOMMENDED PRISMA MODEL
--------------------------------------------------

```prisma
model ContractorMessage {
  id                   String   @id @default(cuid())
  issueId              String
  contractorId         String
  contractorResponseId String?
  direction            String
  channel              String
  messageBody          String
  providerMessageId    String?
  sendStatus           String   @default("sent")
  createdAt            DateTime @default(now())
  updatedAt            DateTime @updatedAt

  issue                Issue               @relation(fields: [issueId], references: [id], onDelete: Cascade)
  contractor           Contractor          @relation(fields: [contractorId], references: [id], onDelete: Cascade)
  contractorResponse   ContractorResponse? @relation(fields: [contractorResponseId], references: [id], onDelete: SetNull)

  @@index([issueId, contractorId, createdAt])
  @@index([contractorResponseId])
}
```

--------------------------------------------------
WHY THIS MODEL IS BEST
--------------------------------------------------

It gives you:
- a durable history of what was sent
- link back to the issue
- link to contractor
- optional link to the specific ContractorResponse that triggered the reply
- ability to support multi-message threading later without rework

==================================================

You already have:

==================================================
4. WHEN TO CREATE ContractorMessage
==================================================

Whenever the user replies from the app.

That means:

File likely involved:
app/api/issues/[id]/reply-to-contractor/route.ts

Current behavior probably:
- validate
- send via Twilio/Resend
- timeline log
- return success

NEW REQUIRED BEHAVIOR:
After successful send:
- create ContractorMessage row

If send fails:
- optionally create ContractorMessage row with sendStatus="failed"
- OR do not create row and just return error

RECOMMENDATION:
Only create row if send succeeds for V1.
This keeps UI cleaner.

==================================================
5. EXACT API BEHAVIOR
==================================================

--------------------------------------------------
Route
--------------------------------------------------

POST /api/issues/[id]/reply-to-contractor

Body should include:
- contractorId
- contractorResponseId (optional but strongly recommended)
- channel
- message

Example:
```json
{
  "contractorId": "abc123",
  "contractorResponseId": "resp_123",
  "channel": "email",
  "message": "Park in the street."
}
```

--------------------------------------------------
Validation
--------------------------------------------------

Validate:
1. issue exists and belongs to logged-in user
2. contractor exists
3. contractor is associated with this issue through dispatch/response
4. message length is valid
5. channel is allowed and contractor has necessary contact info

--------------------------------------------------
Send logic
--------------------------------------------------

If channel === "email":
- send via existing Resend helper

If channel === "sms":
- send via existing Twilio helper

Capture provider message ID if returned.

--------------------------------------------------
After send succeeds
--------------------------------------------------

Create ContractorMessage:

```ts
await prisma.contractorMessage.create({
  data: {
    issueId,
    contractorId,
    contractorResponseId: contractorResponseId ?? null,
    direction: "outbound",
    channel,
    messageBody: message,
    providerMessageId: providerMessageId ?? null,
    sendStatus: "sent",
  },
});
```

--------------------------------------------------
Also log timeline
--------------------------------------------------

Keep your existing timeline log, but improve it.

Example:
eventType = "owner_reply_sent"

payload:
- contractorName
- channel
- messagePreview
- contractorResponseId

==================================================
7. QUERY CHANGES
==================================================

When loading issue detail page, you must include outbound messages.

--------------------------------------------------
If using contractorResponseId link
--------------------------------------------------

For each ContractorResponse, include related ContractorMessage rows.

Example Prisma include idea:

```ts
responses: {
  include: {
    contractor: true,
    dispatch: true,
    messages: {
      where: { direction: "outbound" },
      orderBy: { createdAt: "desc" },
      take: 1,
    },
  },
}
```

Note:
If you do not define relation from ContractorResponse -> ContractorMessage directly,
you can query by:
- issueId
- contractorId
- contractorResponseId

--------------------------------------------------
If simpler for V1
--------------------------------------------------

You can query:
- latest outbound ContractorMessage for same issue + contractor

That is acceptable for V1, especially if you only expect one immediate reply.
