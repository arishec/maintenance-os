MAINTENANCE OS
SMS DISPATCH + INBOUND PROCESSING (FINAL SPEC)

==================================================
PURPOSE
==================================================

This document defines EXACTLY how SMS messaging must work in the system.

This is a CORE product loop:
- send issue → receive replies → compare → choose

If SMS is not implemented correctly, the product breaks.

==================================================
SYSTEM OVERVIEW
==================================================

FLOW:

1. App sends SMS to contractor (via Twilio)
2. Contractor replies to SMS
3. Twilio sends inbound webhook to your API
4. System matches reply to correct dispatch
5. AI parses message
6. System creates ContractorResponse
7. UI updates issue with response

==================================================
1. DATA REQUIREMENTS
==================================================

Each dispatch MUST include:

- id
- issueId
- contractorId
- contractorPhone
- replyToken (UNIQUE)
- status (sent, delivered, replied)
- createdAt

--------------------------------------------------

REPLY TOKEN FORMAT

Use deterministic, human-visible token:

Example:
MNT-7F4K2Q

Rules:
- unique per dispatch
- short (6–8 chars)
- uppercase
- easy to read in SMS

==================================================
2. OUTBOUND SMS (TWILIO)
==================================================

Send from:
- your Twilio number

Message MUST include token.

--------------------------------------------------

EXAMPLE MESSAGE:

New repair request [Ref: MNT-7F4K2Q]

Kitchen sink leak at 123 Main St.
Please reply with:
- estimate
- availability

--------------------------------------------------

WHY TOKEN IS REQUIRED

- enables deterministic matching
- avoids ambiguity
- prevents incorrect assignment

==================================================
3. INBOUND SMS WEBHOOK
==================================================

Endpoint:
POST /api/webhooks/twilio/inbound

Twilio sends:
- From (contractor phone)
- Body (message text)
- MessageSid

--------------------------------------------------

STEP 1 — EXTRACT TOKEN

Parse message body:

Look for:
MNT-[A-Z0-9]+

Example regex:

/MNT-[A-Z0-9]+/

--------------------------------------------------

STEP 2 — MATCH DISPATCH

IF token exists:
→ find dispatch where replyToken == token

IF found:
→ use it

--------------------------------------------------

STEP 3 — FALLBACK (ONLY IF SAFE)

IF NO TOKEN:

Find dispatches where:
- contractorPhone matches
- status is "sent" or "delivered"

IF exactly ONE match:
→ use it

IF multiple matches:
→ DO NOT GUESS
→ mark as unresolved

--------------------------------------------------

STEP 4 — STORE RAW MESSAGE

Always store:
- raw body
- phone
- timestamp
- messageSid

==================================================
4. AI PARSING
==================================================

Send message body to AI (Claude)

Extract:
- price (number)
- availability (text/date)
- notes (optional)

--------------------------------------------------

EXAMPLE INPUT:
"I can do it tomorrow for $200"

OUTPUT:
price: 200
availability: tomorrow

==================================================
5. CREATE CONTRACTOR RESPONSE
==================================================

Create record:

- dispatchId
- issueId
- contractorId
- price
- availability
- rawMessage
- parsedData
- createdAt

--------------------------------------------------

UPDATE DISPATCH:

status = "replied"

==================================================
6. UI UPDATE
==================================================

On issue page:

Show:
- contractor name
- price
- availability
- timestamp

Allow:
- comparison across contractors

==================================================
7. ERROR HANDLING
==================================================

CASE: NO TOKEN + MULTIPLE MATCHES

→ mark as "unresolved"
→ do NOT attach to issue

--------------------------------------------------

CASE: UNKNOWN PHONE

→ ignore OR log

--------------------------------------------------

CASE: DUPLICATE WEBHOOK (TWILIO RETRIES)

→ store messageSid
→ ignore if already processed

==================================================
8. CRITICAL RULES
==================================================

1. NEVER rely on "most recent message"
2. ALWAYS prefer token match
3. ONLY fallback when safe (single candidate)
4. NEVER guess between multiple dispatches

==================================================

TEST 3:
